## <a name="overview"></a>🎯 نظرة عامة على النظام

النظام يدعم نموذج عمل استقدام وتأجير العمالة:

- **الشركة المشغلة (Provider)**: تمتلك الموظفين وتستقدمهم (مثل: شركة Jawad)
- **الشركة العميلة (Client)**: تستأجر موظفين من الشركة المشغلة (مثل: شركة Apple)
- **مدير الفرع (Branch Manager)**: موظف في الشركة العميلة يدير فرع معين

---

## <a name="companies-types"></a>👥 أنواع الشركات والمستخدمين

### 1️⃣ أنواع الشركات (`CompanyTypes`)

```
PROVIDER  = الشركة المشغلة (تمتلك الموظفين)
CLIENT    = الشركة العميلة (تستأجر موظفين)
```

### 2️⃣ أنواع المستخدمين

- **Company Account**: حساب الشركة نفسها (المالك)
- **User Account**: موظف داخل الشركة له صلاحيات محددة
    - موظف عادي
    - مدير فرع (Branch Manager)
    - مدير بصلاحيات خاصة

---

## <a name="statuses"></a>📊 حالات الموظف والتعيين

### حالات الموظف (`EmployeeStatusStatus`)

```
AVAILABLE      = متاح (في الشركة المشغلة، غير معين لأحد)
IN_SERVICE     = في الخدمة (معين لشركة عميلة)
ENDED_SERVICE  = انتهت خدمته
```

### حالات التعيين (`EmployeeAssignedStatus`)

```
PENDING   = بانتظار الموافقة
APPROVED  = موافق عليه
DECLINED  = مرفوض
```

## <a name="full-scenario"></a>🔄 سيناريو كامل: شركة Jawad تعين موظف لشركة Apple

### مثال واقعي:

- **شركة Jawad** (PROVIDER) = شركة استقدام عمالة، لديها موظف اسمه "أحمد محمد"
- **شركة Apple** (CLIENT) = شركة تقنية، لديها فرعين:
    - فرع الرياض (مدير الفرع: خالد)
    - فرع جدة (مدير الفرع: سعد)

#### ⚙️ **الخطوة 1: تسجيل الموظف**

**المستخدم**: مالك شركة Jawad أو موظف لديه صلاحية `view_any_employee`

**المكان**:

```

```

    - الرقم الوظيفي: `JW-1001`

    - الوظيفة
    - تاريخ الميلاد

    - بيانات الراتب (في نموذج الراتب Payroll، ليس في Employee)

2. حفظ الموظف
    - يتم تعيين `company_id = Jawad->id`
    - الحالة الافتراضية: `status = AVAILABLE`
    - `company_assigned_id = null` (غير معين لأحد)

**ملاحظة مهمة**:

```
❗ بيانات الراتب الأساسية (basic_salary, housing_allowance, etc.)
```

## <a name="branch-manager"></a>🎯 صلاحيات ومهام مدير الفرع (خالد)

### 1️⃣ رؤية الموظفين المعينين لفرعه فقط

### ═══════════════════════════════════════

**في صفحة Assigned Employees**:

```php
// خالد يرى فقط موظفي فرع الرياض
$query = Employee::whereHas('branches', function($q) use ($user) {
    $q->where('branches.id', $user->managedBranches()->pluck('id'));
});
```

**مثال**:

- خالد (مدير فرع الرياض): يرى أحمد محمد ✅
- سعد (مدير فرع جدة): لا يرى أحمد محمد ❌

```

├────┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬────┤

│Sta │ - │ - │ P │ P │ P │ O │ O │ P │ P │ P │... │ P │ ?  │

حالات الحضور:
P = Present (حاضر)       ✅
A = Absent (غائب)        ❌

- أيام العمل (P): 24 يوم
- أيام الغياب (A): 1 يوم
- أيام الإجازة (L): 2 يوم
- أيام الراحة (O): 2 يوم
- مستبعد (X): 0 يوم
```

### 6️⃣ إدارة Branch Entries (إدخالات الفرع)

### ═══════════════════════════════════════

**المكان**: `Branch Entries Resource`

**صلاحيات خالد (مدير الفرع)**:

#### ✅ **يمكنه**:

1. **إنشاء إدخال جديد**:

    ```
    - اختيار الفرع: فرع الرياض فقط
    - اختيار الموظف: من موظفي فرعه
    - نوع الإدخال: overtime / deduction / addition
    - المبلغ، السبب، الوصف
    - الحالة: تُحفظ كـ DRAFT
    ```

2. **تعديل إدخال Draft أو Rejected**:

    ```
    - يمكنه تعديل الإدخالات بحالة:
      ✅ DRAFT
      ✅ REJECTED (بعد رفضها من المالك)
    ```

3. **إرسال الإدخال (Submit)**:

    ```
    - تغيير الحالة من DRAFT → SUBMITTED
    - يُسجل اسمه في submitted_by
    - يُسجل توقيت الإرسال في submitted_at
    ```

4. **حذف إدخال Draft فقط**

#### ❌ **لا يمكنه**:

1. **اعتماد الإدخال (Approve)** - فقط مالك الشركة
2. **رفض الإدخال (Reject)** - فقط مالك الشركة
3. **رؤية أو تعديل إدخالات الفروع الأخرى**
4. **تعديل أو حذف إدخال معتمد (APPROVED)**

**مثال على workflow**:

```
┌─────────────────────────────────────────────────────────────┐
│ خالد ينشئ إدخال → DRAFT                                    │
│          ↓                                                  │
│ خالد يضغط Submit → SUBMITTED                               │
│          ↓                                                  │
│ مالك Apple يراجع:                                          │
│     → Approve ✅ → APPROVED (يظهر في Payroll)              │
│     → Reject ❌ → REJECTED (يعود لخالد للتعديل)            │
│          ↓                                                  │
│ خالد يعدل ويعيد Submit → SUBMITTED (مرة أخرى)             │
└─────────────────────────────────────────────────────────────┘
```

**الكود**:

```php
// في BranchEntryResource
protected static function isBranchManager(): bool
{
    $user = Filament::auth()->user();
    return $user instanceof User && $user->isBranchManager();
}

// Action: Submit
->visible(fn (BranchEntry $record) =>
    in_array($record->status, [
        BranchEntryStatus::DRAFT,
        BranchEntryStatus::REJECTED
    ])
)

// Action: Approve (مخفي عن مدير الفرع)
->visible(fn (BranchEntry $record) =>
    $record->status === BranchEntryStatus::SUBMITTED
    && !static::isBranchManager()
)
```

## 📌 ملاحظات مهمة جداً

### 1️⃣ بيانات الراتب

```
⚠️ تخزن في جدول Payroll وليس Employee
   - basic_salary
   - housing_allowance
   - transportation_allowance
   - food_allowance
   - other_allowance
   - fees
   - total_package
```

### 2️⃣ التعيين الافتراضي بحالة PENDING

```
⚠️ عند Assign من Provider:
   status = PENDING (ليس APPROVED)

   فقط عند موافقة Client:
   status = APPROVED
```

### 3️⃣ اختيار الفرع إلزامي

```
⚠️ لا يمكن الموافقة على تعيين بدون اختيار فرع
   نافذة منبثقة إلزامية مع required validation
```

### 4️⃣ معادلة العمل الإضافي الجديدة

```
⚠️ التحديث الأخير (2026-02-23):
   overtime = (total_salary / 240 * hours)
            + (basic_salary / 480 * hours)

   القديم (محذوف):
   overtime = sum('amount') من جدول overtimes
```

### 5️⃣ قواعد الخصومات الجديدة

```
⚠️ التحديث الأخير (2026-02-23):

   A (Absent) = خصم راتب فقط (بدون رسوم):
   deduction = absent_days * (total_salary / 30)

   L/O/X = خصم راتب + رسوم:
   deduction = days * ((total_salary + fees) / 30)
```

### 6️⃣ الخصومات محظورة الآن

```
⚠️ التحديث الأخير (2026-02-23):
   ❌ DeductionResource مخفي من Provider
   ❌ تبويب Deductions مخفي من Provider في EmployeeEntries
   ✅ فقط Client يمكنه إدخال الخصومات
   ❌ مديري الفروع لا يمكنهم إدخال خصومات
```

### 7️⃣ تاريخ التعيين قابل للتعديل

```
⚠️ التحديث الأخير (2026-02-23):
   ✅ مالك Client يمكنه تعديل start_date
   Action: Edit Start Date
   يحدث أيضاً في branch_employee pivot
```
